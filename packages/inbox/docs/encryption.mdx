# Encryption

Inbox is end-to-end encrypted. The server only ever sees ciphertext, envelope metadata for routing, and timing. This page documents the cryptographic primitives, where the keys live, and the rotation / recovery flows.

## Primitives

| Operation | Algorithm | Library |
|-----------|-----------|---------|
| Symmetric encryption | AES-256-GCM | Web Crypto (`crypto.subtle`) |
| Key derivation from recovery phrase | PBKDF2-HMAC-SHA256, 600 000 iterations | Web Crypto |
| Random nonces / IDs | `crypto.getRandomValues` | Web Crypto |
| Public-key signing | secp256k1 ECDSA via `KeyManager` | `@oxyhq/core` |

No third-party crypto libraries — we lean exclusively on Web Crypto so the browser's underlying implementation is the trust boundary.

## Key hierarchy

```
Recovery phrase (BIP-39, 12 words)
   │
   ▼ PBKDF2 (salt = account id, iter = 600_000)
Recovery master key (256-bit)
   │
   ▼ AES-KW (key wrapping)
Account encryption key (AES-256-GCM CryptoKey, non-extractable)
   │
   ▼ derives via HKDF (info = "msg")
Per-message key  ──▶ ciphertext + 12-byte nonce + auth tag
```

- The **recovery phrase** lives only in the user's head (or password manager). It is shown once at identity creation and never sent to the server.
- The **recovery master key** is derived on demand for key unwrapping; it is never persisted.
- The **account encryption key** is created with `extractable: false` and stored in IndexedDB. It cannot be exported, copied, or read out by JS — only used as a handle into `crypto.subtle.encrypt` / `decrypt`.
- **Per-message keys** are derived per send/receive so a single key compromise doesn't cascade.

## Where keys live

| Storage | What's there |
|---------|--------------|
| Memory | Decrypted message body for the active thread (cleared on navigation). |
| IndexedDB | Non-extractable `CryptoKey` handles + wrapped key blobs. |
| Server | Wrapped account encryption key (encrypted under the recovery master). Used only during multi-device recovery. |
| Recovery phrase | The user. Source of truth — losing this and all signed-in devices destroys the mailbox permanently. |

Nothing ever lands in `localStorage`, `sessionStorage`, or cookies.

## New device flow

1. User signs in on a new device with their recovery phrase.
2. Client derives the recovery master key from the phrase.
3. Server returns the wrapped account encryption key.
4. Client unwraps the AES key with `crypto.subtle.unwrapKey({ name: 'AES-KW' }, …, { extractable: false })`.
5. The unwrapped key lands in IndexedDB and the recovery master is wiped from memory.

## Sending a message

```ts
async function encryptBody(plaintext: string, accountKey: CryptoKey) {
  const nonce = crypto.getRandomValues(new Uint8Array(12));
  const ciphertext = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv: nonce },
    accountKey,
    new TextEncoder().encode(plaintext),
  );
  return { ciphertext: new Uint8Array(ciphertext), nonce };
}
```

The subject line is encrypted with the body so the server can't fingerprint by subject either. Only the envelope (sender, recipient, timestamp, size) is plaintext for routing.

## Search

Because the server can't read message bodies, full-text search runs **client-side** against the locally cached, decrypted index. The index is rebuilt incrementally as messages arrive and stored encrypted in IndexedDB.

## Key rotation

- **Per-message keys** rotate every send (derived fresh from HKDF).
- **Account encryption keys** rotate on explicit user action ("rotate encryption key"). The new key is wrapped under the same recovery master; existing messages are re-encrypted lazily on read and migrated on a background job.
- **Recovery master keys** only change if the user changes their recovery phrase — which forces a full re-wrap of the account key.

## Threats this does NOT defend against

- A compromised browser extension with access to `window` — JS can read the decrypted body once it's in DOM.
- Backdoored hardware / OS — we trust the browser sandbox.
- Forgetting the recovery phrase + losing every signed-in device — the data is gone. There is no server-side reset.

## Audit trail

The encryption module lives in `packages/inbox/src/crypto/`. It's small, single-file per primitive, and intentionally easy to review. PRs that touch it require a security review label before merge.
