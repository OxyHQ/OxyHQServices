# Offline & sync

Inbox treats the network as optional. The user can read, search, draft, archive, and label messages with the laptop unplugged, the phone in airplane mode, the radio off — and every action is reconciled the moment connectivity returns.

## Storage layers

| Layer | Where | Holds |
|-------|-------|-------|
| Persisted query cache | IndexedDB via `@tanstack/react-query-persist-client` | Hydrated inbox lists, threads, search snapshots |
| Mutation queue | IndexedDB | Pending sends, deletes, label changes, drafts |
| Encrypted body cache | IndexedDB | Decrypted bodies + attachments, keyed by message id |
| Key store | IndexedDB (non-extractable) | Per-account AES key, RSA / Ed25519 key pair |
| Service Worker precache | Cache Storage API | App shell (HTML, JS, CSS, fonts) |

All four are namespaced by `accountId`. Switching accounts swaps the active namespace and the UI seamlessly re-renders the cached state for the new account.

## Reads

Reads are query-cached:

- Inbox lists (`/inbox/threads`, paginated) are stored with `staleTime: 5min`.
- Thread bodies are stored with `staleTime: Infinity` and only refetched on explicit pull-to-refresh, since message bodies are immutable.
- Labels and counts are stored with `staleTime: 30s` and refetched on focus.

When the user opens a thread offline, the UI serves the cached decrypted body instantly; no spinner, no error. If the body was never cached, the UI shows an offline placeholder with the envelope metadata and a **Retry when online** action.

## Writes

Every mutation hook (`useSendMessage`, `useArchiveThread`, `useApplyLabel`, …) uses:

```ts
useMutation({
  networkMode: 'offlineFirst',
  mutationKey: ['inbox', 'send', draftId],
  // ...
});
```

`networkMode: 'offlineFirst'` means:

- If online, the mutation runs immediately, and React Query updates the cache via `onMutate` (optimistic) and `onSuccess` (reconcile).
- If offline, the mutation is paused — `onMutate` still fires (so the UI updates immediately) but the network call is deferred.
- When connectivity returns, paused mutations are replayed in order. Each mutation runs against the freshest cache, so an "archive" followed by an "unarchive" replays as the user expected.

The persisted cache keeps the queue across reloads. Closing the tab, restarting the device, or losing battery does not lose pending work.

## Conflict resolution

Three conflict classes show up in practice:

1. **Last-writer-wins on labels.** If two devices apply different labels to the same thread, both labels survive — labels are a set, not a single value. The "archived" label is special: an explicit `unarchive` cancels a pending `archive`.
2. **Server-authoritative on delivery state.** A draft that the server reports as already-sent (because another device sent it) cancels the pending `send` locally.
3. **Body integrity.** Encrypted bodies are immutable; a body re-fetch will only deserialise if the ciphertext matches the `messageId`'s declared hash.

The conflict-resolution code lives in `useInboxSync` and runs on reconnect.

## The `useOnlineStatus` hook

The whole UI knows when the network is down via a single `useOnlineStatus()` hook backed by `useSyncExternalStore` over React Query's `onlineManager`. The app uses it for:

- A persistent offline banner anchored to the top of the screen.
- A muted "Syncing…" indicator next to the account avatar while the mutation queue drains.
- Greyed-out send buttons on attachments larger than the per-tier offline cap (so we do not silently queue a 50 MB upload that will fail on reconnection).

You can use the same hook in app code:

```tsx
import { useOnlineStatus } from '@oxyhq/services';

function OfflineBanner() {
  const online = useOnlineStatus();
  if (online) return null;
  return <p className="bg-amber-100">You are offline. Changes will sync when you reconnect.</p>;
}
```

## Service Worker

The Service Worker only caches the app shell. It deliberately does not cache API responses — they would bypass the encryption layer and end up in plain Cache Storage, which is readable by other site code. Encrypted bodies live in IndexedDB and are decrypted on demand inside the React tree.

Service Worker updates roll out atomically: the new version installs in the background, waits for every tab to close, and activates on next load. There is no banner urging the user to refresh — Inbox is durable across version changes.
