# Security

The MyAccount security center is where the user sees and controls everything authentication-related: their active sessions, registered devices, 2FA, recovery phrase health, and the platform's signed activity log.

## Sessions

Every active sign-in is a session. `/(tabs)/security` lists them — current device on top, the rest sorted by last activity.

For each session the UI shows:

- Device model, OS, app version (from the `User-Agent` parsed server-side).
- Approximate location (city/region from IP-to-geo, never the IP itself).
- Last seen timestamp and a short summary of recent activity.
- A **Sign out** button for any session except the current one. Tapping it calls `DELETE /sessions/:id` and the target session receives a `session_removed` event over its open socket and signs out locally within a second.
- A **Sign out everywhere** action revokes every session except the current one in a single batched call.

Sessions are rotated automatically: every successful refresh issues a new session id, and the previous id is hard-revoked. A leaked refresh token only works until the legitimate device refreshes — at which point the leaked token becomes useless and a `session_replay_detected` event fires.

## Devices

A device is distinct from a session — devices outlive sessions and survive sign-outs. The Devices list shows every device that has ever signed in to the account, with:

- Friendly name (editable; defaults to the model).
- Trust state — `trusted`, `pending`, or `untrusted`.
- Cryptographic fingerprint (the device's public key, abbreviated).
- Last sign-in time and IP region.

Marking a device `untrusted` revokes its refresh token without ending the active session; the next refresh fails and the device signs out. Use this when a phone has been lost or sold.

## Two-factor auth

MyAccount supports TOTP and WebAuthn / passkeys. Both are managed under `/settings/security/2fa`.

- **TOTP** — scan a QR code in any authenticator app (Aegis, 1Password, Authy, Google Authenticator). On enrolment the UI presents 10 single-use recovery codes; the user must check **I have saved my codes** before completing setup.
- **Passkeys** — uses platform WebAuthn. On supported browsers/OSes the user can register Face ID / Touch ID / Windows Hello / a hardware key. Passkeys are bound to the device's keychain or the user's iCloud / Google account, depending on platform.

Either factor alone is enough to satisfy 2FA at sign-in. Enabling both gives the user a fallback if one fails (lost phone vs lost yubikey).

## Recovery phrase health

The Security screen surfaces the recovery phrase state with a banner:

| State | Means | Action |
|-------|-------|--------|
| **Acknowledged** | User has clicked through the recovery-phrase confirmation screen. | None. |
| **Not acknowledged** | Identity exists but the user has never confirmed seeing the phrase. | Persistent reminder, with a CTA to view the phrase. |
| **Identity missing** | No identity on the device (post sign-in but pre-create). | Block sensitive actions, route to `/(auth)/create-identity`. |

`KeyManager.hasIdentity()` is the source of truth for the "identity missing" branch. It requires both keys present, well-formed, and matching, not just key files on disk — so a corrupted or partially-restored identity is correctly flagged.

## Activity log

`/(tabs)/activity` is an infinite-scroll list of every security-relevant event the platform has recorded for the account:

| Event | Triggered by |
|-------|--------------|
| `session.created` | Successful sign-in |
| `session.refreshed` | Token refresh |
| `session.revoked` | Manual sign-out or remote revocation |
| `password.changed` | Self-service password change |
| `password.reset` | Recovery flow completed |
| `2fa.enabled` / `2fa.disabled` | 2FA enrolment / removal |
| `device.trusted` | Trust state change |
| `recovery_phrase.viewed` | User opened the recovery-phrase screen |
| `account.deleted` | Self-service deletion accepted |

Each event carries a server-signed `audit` field — the SDK verifies the signature before rendering, so a compromised local cache cannot spoof entries.

## Delete account

Account deletion is irreversible by design. The flow guards against accidents:

1. The user opens **Settings → Delete account**.
2. The screen lists every consequence (federated content lost, key material destroyed, recovery impossible).
3. The user types their own username to confirm.
4. The client signs the deletion intent with the on-device identity key.
5. `KeyManager.hasIdentity()` runs as a pre-flight — if the identity is missing, deletion is blocked (we cannot prove the user controls the account).
6. The platform schedules a 30-day hard-delete with a "Cancel" link emailed to the account's verified address. After 30 days, everything is gone for good.
