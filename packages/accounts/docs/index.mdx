# Accounts (MyAccount)

The Oxy MyAccount app is the canonical surface for managing an existing account: devices, sessions, privacy, language, payment methods, connected apps, and security activity. Built on Expo Router + `@oxyhq/services`, it ships as a native iOS/Android app and a web build, all from the same source.

**Keyless by design**: identity creation, key management, and recovery phrases live in Commons by Oxy (`packages/commons`). `accounts.oxy.so` is the sole owner of account management in the ecosystem — the IdP (`auth.oxy.so`) permanently redirects its former settings pages here.

## What it is

A user-facing app, not a library. Most integrators won't embed it — they'll deep-link into it from their own surfaces. The app is open source so the patterns it uses are visible and reusable.

## Highlights

- **11 locales** — English + Spanish fully populated, 9 partial. Device locale resolved via `Intl.DateTimeFormat().resolvedOptions().locale` (no `expo-localization` native module).
- **Typed Expo Router routes** — `typedRoutes: true` in `app.json`; every `router.push()` uses a typed path. No `as any`.
- **Error boundaries** — at root, `(tabs)`, and `(auth)` levels using a shared `ErrorFallback`.
- **Activity history** — `/(tabs)/activity.tsx` pulls from `GET /security/activity` with infinite scroll.
- **Connected apps** — list and revoke OAuth grants for third-party applications.
- **Delete account** — signed deletion with username confirmation, followed by a strict local identity purge and sign-out-all.

## Sign-in

Accounts is a standard official relying party on the device-first session model:

- On cold boot, `OxyProvider` restores the session silently from the server-side device session (see [device sessions](../../../docs/auth/device-session.md)). It never auto-redirects to a login page.
- Logged-out users get the SDK sign-in surface (`OxyAccountDialog`): Commons QR / deep-link first, password collapsed under "Sign in without the app".
- The app implements no local session restore, callbacks, or token plumbing — the SDK owns all of it.

## Deep-linking into MyAccount

Most apps integrate by sending the user to MyAccount to perform a specific action:

```ts
// Open the security screen
Linking.openURL('https://accounts.oxy.so/security');
```

Routes the app surfaces externally:

| URL | Purpose |
|-----|---------|
| `/` | Account overview |
| `/personal-info` | Profile fields |
| `/security` | Security & devices |
| `/sessions` | Active sessions |
| `/activity` | Security activity log |
| `/payments` | Payments, subscription, wallet |
| `/data` | Data & privacy |
| `/authorize` | Connected apps (OAuth grants) |

## Internal patterns worth borrowing

These apply to any Oxy-flavoured Expo app:

- **`useUpdateProfile` for username writes** — `useUpdateProfile().mutateAsync()` gets the optimistic update + cache invalidation; calling `oxyServices.updateProfile()` directly skips both.
- **Stable lazy initial values** — `useState(() => initial)` for form state initialised from a query, so a remount doesn't reset to stale data.
- **Session-keyed routing** — `(auth)`↔`(tabs)` keys purely on `isAuthResolved` / `isAuthenticated`; the root `<Stack>` is the sole authority for the group swap.
- **`useUpdatePrivacySettings`** — don't call `invalidateAccountQueries(queryClient)` in `onSuccess` (defeats optimistic merge). Use `{ ...previous, ...requested, ...incoming }` merge in `onMutate`, targeted invalidate in `onError`.

## Test coverage

Jest tests live in `packages/accounts` and run with the package's own `bun run test` script (ts-jest — never Bun's native test runner).

## API surface

There is no public JS API for the app — it's a deployed product. For programmatic equivalents, integrate via [`@oxyhq/core`](/developers/docs/core) (Node/server) or [`@oxyhq/services`](/developers/docs/services) (Expo/RN/web).
